# MCP Control Center 项目整体架构与模块关系

## 1. 产品定位

MCP Control Center 是面向企业内部研发、产品、运维和平台管理员的本地优先 MCP 管控台。它管理的不是单个 MCP 进程，而是由 Workspace、Service、MCP、工具契约、调用记录、错误记录、基础设施和知识文档组成的完整资产链路。

首版使用结构化演示数据，但所有页面按照真实后台可替换的 API 结构设计；数据变化只发生在浏览器内，用于验证企业用户的管理路径。

## 2. 核心对象关系

```text
Workspace
├── Service
│   ├── MCP
│   │   ├── MCP 版本
│   │   ├── 工具契约 / Prompt / Resources
│   │   ├── 调用记录
│   │   ├── 错误记录 / 调用上下文
│   │   └── 测试与鉴权配置
│   └── Service 基础设施 / 告警 / 沙盒
├── Workspace 可见性与环境配置
└── 成员、角色与权限

调用分析 ─────┐
错误中心 ─────┼──> MCP 详情（对应 Tab）──> 调用上下文 / 版本 / 修复操作
服务维护 ─────┘

MCP 市场 ──> 接入详情弹窗 ──> 沙盒安装 ──> 安全审核 ──> Workspace 上架
知识库 ────> 文档详情 / 解析结果 ──> 关联 Service、MCP、错误码
上架管理 ──> 基础配置 ─> 技术配置 ─> 沙盒验证 ─> 审批上架 ─> 发布结果
权限管理 ──> 模块权限矩阵 ──> 角色 / 成员 ──> 查询、编辑、处理能力
```

## 3. 左侧主导航与跳转约定

| 模块 | 默认入口 | 主要跳转 |
| --- | --- | --- |
| 总览 | `/overview` | 指标卡进入 MCP 列表、调用分析、错误中心；排行进入 MCP 详情对应 Tab |
| Service 管理 | `/services` | Service 行展开抽屉；查看详情进入 Service 详情页；下属 MCP 进入 MCP 详情 |
| MCP 列表 | `/mcps` | 搜索、Workspace、Service、状态、负责人筛选；行进入 MCP 详情 |
| 调用分析 | `/analytics` | 日期筛选、排序、调用记录；MCP 行进入 MCP 详情调用分析 Tab |
| 错误中心 | `/errors` | 日期、原因、解决时长筛选；告警进入 MCP 报错分析 Tab；日志进入上下文弹窗 |
| 上架管理 | `/onboarding` | 五步向导：基础配置、技术配置、沙盒验证、审批上架、发布结果 |
| 服务维护 | `/maintenance` | 服务配置、告警、风险日志、异常详情、Ping 测试、Workspace 环境 |
| MCP 市场 | `/market` | 分类、协议、鉴权、供应商筛选；条目进入接入详情弹窗和沙盒安装 |
| 知识库 | `/knowledge` | 文档搜索、上传、解析、编辑、删除、详情、关联对象 |
| 权限管理 | `/permissions` | 角色权限矩阵、成员授权、模块查询/编辑/处理权限 |

## 4. 全局交互规则

1. 顶部 Workspace 切换会影响 MCP 列表、Service 列表、上架管理、维护环境和市场安装目标。
2. 所有指标卡必须有明确去向；不能只展示数字。
3. 所有列表都支持搜索、筛选、排序、空状态和清除条件。
4. 编辑动作必须显示当前权限；没有编辑权限时显示只读状态和申请权限入口。
5. 日志、错误、文档和接入项优先使用弹窗或抽屉展示上下文，关闭后回到原列表。
6. 所有演示写操作通过 Toast 或状态字段反馈，并能在当前页面看到结果变化。

## 5. API 预留边界

原型组件只依赖领域数据和动作，不直接绑定数据库。后续可替换以下接口：

- `GET /api/workspaces`
- `GET /api/overview?workspaceId=&range=`
- `GET /api/services?workspaceId=`
- `GET /api/mcps?workspaceId=&serviceId=&status=`
- `GET /api/analytics/calls?from=&to=&sort=`
- `GET /api/errors?from=&to=&reason=&sort=`
- `GET /api/mcps/:id/context/:traceId`
- `POST /api/onboarding/validate`、`POST /api/onboarding/publish`
- `POST /api/maintenance/ping`
- `GET/POST/PATCH/DELETE /api/knowledge/documents`
- `GET/PATCH /api/permissions`
